# PM-401 恢复与失败成本优化详细设计

版本：v0.1 初稿

## 设计结论

PM-401 不做新系统、不扩新内容、不重做界面。最小可行方案是把失败成本调到“疼但不重开”，把结算页和洞府消息讲清楚，并用现有休息、丹药、研读、技能配置入口闭环。

已制作的界面部分：

- `scenes/lilian/expedition_result.tscn` 底部规则说明改为战败后的下一步提示。

代码和配置交给实现同学，本文只给目标、变量和验收。

## 玩家体验目标

- 战败后玩家知道：丢了什么、还剩什么、下一步点哪里恢复。
- 一次失败损失可在 `1-3` 次普通历练或 `1-2` 次休息内恢复。
- 早期一次短历练应能支撑至少一炉基础丹药的材料需求。
- “没有新功法/技能学习”的感受先用路径提示解决：典籍掉落、研读、技能配置要被看见。

## 当前问题定位

| 问题 | 现象 | 当前位置 | PM-401 处理 |
|---|---|---|---|
| 战败像重开 | 结算知道失败，但不清楚损失上限和恢复入口 | `expedition_result` | 强化结算说明，轻调战败惩罚 |
| 伤势与休息关系不清 | 洞府有休息，但玩家不知道能恢复多少 | `cave_hub` / `GameState.rest()` | 洞府消息补“休息减伤势”，数值用现有 `injury_recovery=2` |
| 丹药定位不清 | 疗伤丹描述像治伤势，当前主要是气血恢复 | `data/item.yaml` / 物品提示 | 二选一：改描述，或接入 `injury` 效果 |
| 历练资源偏少 | 有时材料不足以开一炉 | `locations` / `expedition_common_events` / `reward_budget` | 调早期采集和短历练材料下限 |
| 学习路径不明显 | 新功法/技能不是没有，而是入口和掉落目标不明显 | 洞府底部“研读/技能” | 结算/失败提示指向研读与技能配置 |

## 核心循环

```text
外出历练 -> 获得材料/典籍/战斗经验
  -> 失败或主动返程 -> 结算看清得失
  -> 洞府休息/炼丹/研读/技能配置
  -> 再次出门，压力下降或目标更明确
```

## 机制规格：战败成本

目的：保留风险感，但避免一次失败删除一段游玩。

玩家感受：输了会痛，但能立刻看到“我还保住了大部分收获，回去休息再来”。

输入：

- 战斗失败，`exit_reason = defeated`。
- 本次历练 `loot`、最终 `hp/mp`、当前 `injury_days`。

输出：

- 掉落本次历练部分战利品，不动存档旧库存。
- 追加伤势，保留气血下限。
- 结算页显示“获得道具 / 战败掉落 / 下一步恢复”。

建议调参：

| 变量 | 当前 | 建议 | 理由 |
|---|---:|---:|---|
| `defeat_loot_drop_ratio` | `0.30` | `[PLACEHOLDER] 0.20` | 保住 80% 本次收获，失败仍有推进 |
| `defeat_injury_days` | `3` | `[PLACEHOLDER] 2` | 当前休息一次减 2 日，战败后一次休息即可清伤 |
| `defeat_hp_floor_ratio` | `0.25` | `[PLACEHOLDER] 0.40` | 玩家不休息也不会处在一碰就倒的状态 |
| `fled_injury_days` | `1` | `1` | 战中遁走已有轻成本，先不动 |

边界：

- 无战利品时不显示空的“战败掉落”。
- 掉落只作用于本次 `loot`，禁止扣旧背包。
- Boss / 精英不额外叠惩罚，难度由战斗本身承担。

失败状态：

- 如果玩家连续两次普通战败，说明不是失败成本问题，而是地点推荐、敌人数值或自动战斗配置问题，移交 PM-405 回归处理。

## 机制规格：恢复路径

目的：让恢复是一次明确决策，不是玩家猜系统。

现有恢复源：

| 来源 | 当前能力 | PM-401 表达 |
|---|---|---|
| 洞府休息 | 气血/法力回满，伤势减少 `2` 日 | 战败结算和洞府消息都提示 |
| 历练丹药槽 | 回气/疗伤/回灵等恢复 HP/MP | 战败前准备方向，不作为新系统 |
| 恢复事件 | 低资源时更容易抽到 recover | 保持，不新增复杂保底 |
| 修炼经过天数 | 修炼会缓慢减伤，但收益受伤势折半 | 提醒玩家先休息再闭关 |

疗伤丹处理二选一，优先级从懒到重：

1. 最小配置修正：把疗伤丹描述从“治疗伤势”改成“恢复气血”，避免误导。
2. 完整修正：给疗伤丹 `use_effect: injury` 并让 `GameState.use_inventory_item()` 支持消耗品使用。这个是新行为，PM-401 可做但要补测试。

## 机制规格：基础资源产出

目的：一次早期短历练至少能开一炉基础丹，不让恢复循环断粮。

材料门槛：

- 回气丹：`灵草 x2`。
- 回灵丹：`灵草 x1 + 灵果 x1`。
- 疗伤丹：`灵草 x2 + 灵果 x1`。

建议目标：

| 活动 | 当前参考 | PM-401 目标 |
|---|---|---|
| 早期采集节点 | `gather_herb_early` 目标草药 `4` | 保持或提高权重，单节点可开至少一炉回气丹 |
| 早期短历练 | `short_expedition_early` 目标草药 `6` | 结算均值覆盖 `2-3` 炉基础丹的核心草药 |
| 战败后短历练 | 失败损失目标 `2` 分钟 | 仍能带回至少一半材料，不让失败归零 |

配置实现建议：

- 优先调 `data/didian.yaml` 早期地点奖励池和 `data/lilian_common_events.yaml` 采集权重。
- 其次调 `data/lilian_rules.yaml > reward_budget.daily_base_value`。
- 不新增“必掉保底”代码，除非固定种子测试仍出现连续两次无基础材料。

## 机制规格：功法与技能学习提示

目的：不新增功法/技能，先解决“我不知道哪里学、哪里装”的认知问题。

玩家路径：

1. 历练或事件获得典籍。
2. 回洞府点底部「研读」学习。
3. 点底部「技能」查看已学技能。
4. 需要战斗配置时进入属性/配置面板调整槽位。

界面文案建议：

| 位置 | 文案目标 |
|---|---|
| 战败结算 | “先休息，再去研读典籍或配置技能；缺丹药去炼丹。” |
| 洞府上次历练消息 | 若上次战败：显示“建议休息/炼丹/调整技能后再出发” |
| 典籍物品提示 | 明确“在洞府底部「研读」使用” |

## UI 设计

视觉方向：修仙洞府 UI，旧纸、土黄、深褐描边，低动效，不遮挡主要内容。

已落地：

- 结算页底部规则条从泛泛结算说明改成恢复指引。
- 不新增按钮，不新增弹窗，不改结算布局。

后续可做但不在本次直接做：

- `RuleLabel` 改为脚本动态文本：战败、主动返程、战中遁走各一套。
- 洞府 `MessageLabel` 根据 `last_lilian_summary.exit_reason` 显示恢复建议。
- 若做动态文本，相关节点已 `unique_name_in_owner`，脚本用 `%RuleLabel` / `%MessageLabel`。

## 实现拆分

代码任务：

| 文件 | 任务 | 验收 |
|---|---|---|
| `scripts/lilian/expedition_result.gd` | 渲染动态下一步提示 | 战败/主动返程文案不同 |
| `scripts/sim/cave_hub.gd` | 上次历练为战败时提示休息、炼丹、研读/技能 | 回洞府第一眼看到解法 |
| `scripts/sim/game_state.gd` | 仅当选择完整修正时，支持消耗品 `hp/mp/injury` 直接使用 | 疗伤丹能真实缩短伤势 |
| `tests/run_lilian_tests.gd` | 更新战败掉落和伤势期望 | 固定种子通过 |
| `tests/run_simulation_tests.gd` | 若接入疗伤丹治伤，补一条消耗品测试 | 伤势减少且物品扣除 |

配置任务：

| 文件 | 任务 | 验收 |
|---|---|---|
| `data/lilian_rules.yaml` | 调战败掉落、伤势、气血下限 | 失败恢复周期 `1-3` 次普通历练 |
| `data/didian.yaml` | 提高早期地点基础材料稳定性 | 短历练均值可炼至少一炉 |
| `data/lilian_common_events.yaml` | 调早期采集/恢复事件权重或奖励 | 低资源时不连续卡死 |
| `data/item.yaml` | 修正疗伤丹描述，或增加 `injury` 效果 | 描述与实际效果一致 |
| `data/pingheng_design/reward_loop.csv` | 记录新目标值 | PM-405 能按表复测 |

## 验收清单

- 战败结算能看见保留收获、掉落损失和恢复建议。
- 战败后点一次休息即可清掉 PM-401 建议伤势值。
- 新局早期完成一次短历练，至少能凑齐一炉回气丹材料。
- 结算、洞府、物品说明不再暗示不存在的治疗路径。
- 未新增全屏界面、未新增系统入口、未改 P5 内容范围。
